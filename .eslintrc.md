# ESLint & Prettier Configuration

This project uses ESLint and Prettier for code quality and formatting.

## Setup

The following packages are installed:

- `@nuxt/eslint` - Nuxt's official ESLint integration
- `eslint` - Core ESLint package
- `prettier` - Code formatter
- `eslint-config-prettier` - Disables ESLint rules that conflict with Prettier
- `eslint-plugin-prettier` - Runs Prettier as an ESLint rule
- `@typescript-eslint/eslint-plugin` - TypeScript-specific linting rules
- `@typescript-eslint/parser` - Allows ESLint to parse TypeScript

## Configuration Files

### `.prettierrc`

Prettier configuration with project-specific formatting rules:

- No semicolons
- Single quotes
- 2-space indentation
- No trailing commas
- 100 character line width

### `eslint.config.mjs`

ESLint configuration using the flat config format (ESLint 9+):

- TypeScript rules for better type safety
- Vue-specific rules for component structure
- Custom rules for console usage and code style

### `.vscode/settings.json`

VSCode workspace settings for automatic formatting:

- Format on save enabled
- ESLint auto-fix on save
- Prettier as default formatter

## Available Scripts

### Linting

```bash
# Check for linting issues
npm run lint

# Fix auto-fixable linting issues
npm run lint:fix
```

### Formatting

```bash
# Format all files
npm run format

# Check if files are formatted correctly (CI/CD)
npm run format:check
```

### Type Checking

```bash
# Run TypeScript type checking
npm run typecheck
```

## Pre-commit Hook (Optional)

Consider adding a pre-commit hook using `husky` and `lint-staged` to automatically lint and format staged files before committing:

```bash
npm install -D husky lint-staged
npx husky init
```

Add to `package.json`:

```json
{
  "lint-staged": {
    "*.{js,ts,vue}": ["eslint --fix", "prettier --write"],
    "*.{json,md,scss,css}": ["prettier --write"]
  }
}
```

## Rules Overview

### TypeScript Rules

- `@typescript-eslint/no-unused-vars`: Warns about unused variables (except those prefixed with `_`)
- `@typescript-eslint/no-explicit-any`: Warns about using `any` type (prefer `unknown` or specific types)

### Vue Rules

- `vue/multi-word-component-names`: Disabled (allows single-word component names)
- `vue/no-multiple-template-root`: Disabled (Vue 3 supports fragments)
- `vue/html-self-closing`: Enforces proper self-closing tags
- `vue/max-attributes-per-line`: Limits attributes per line for readability

### General Rules

- `no-console`: Warns about console usage (except `console.warn` and `console.error`)
- `prefer-const`: Warns when `let` is used but variable is never reassigned
- `no-unused-vars`: Disabled in favor of TypeScript's rule

## Best Practices

1. **Run linting before committing**: Always run `npm run lint` before committing changes
2. **Fix issues promptly**: Don't accumulate linting warnings/errors
3. **Use TypeScript properly**: Avoid `any` types, prefer specific types or `unknown`
4. **Format your code**: Use `npm run format` or enable format-on-save in your IDE
5. **Keep configuration updated**: Review and update linting rules as the project evolves
